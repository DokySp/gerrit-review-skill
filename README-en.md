# Gerrit Review Skill

This Codex skill queries Gerrit patch set reviews, validates AI reviewer comments against the actual code, and prepares Gerrit reply drafts, draft comments, or real review comments.

The default behavior is read-only analysis and dry-run output. Drafts or published comments are created only when the user explicitly asks for them.

This skill requires local Gerrit account configuration before use. If the config is missing, scripts stop before querying Gerrit, creating drafts, or publishing comments, and print setup guidance instead.

## Core Features

- Collect local git context from `HEAD`, `Change-Id`, remotes, and branches.
- Create and inspect local Gerrit account configuration.
- Diagnose Gerrit SSH connectivity.
- Query Gerrit change and patch set metadata.
- Extract AI reviewer inline comments by file and line.
- Inspect the target code and call paths behind each comment.
- Classify each comment as `valid`, `partially_valid`, `false_positive`, or `needs_more_context`.
- Draft Gerrit replies for comments classified as `false_positive`.
- Prefix every Gerrit comment written by Codex with `[Codex로 생성됨]`.
- Build Gerrit `ReviewInput` JSON for real review comments.
- Generate dry-run or executable REST draft comment requests when REST authentication is available.

## Default Workflow

1. Check whether Gerrit account config exists.
2. Stop and print setup guidance if config is missing.
3. Collect local git context.
4. Check Gerrit SSH access.
5. Query Gerrit by current `HEAD`, `Change-Id`, or a user-provided change number.
6. Verify that Gerrit current patch set revision matches local `HEAD`.
7. Extract AI reviewer inline comments.
8. Validate each comment against the real code and call path.
9. Draft Gerrit replies.
10. Generate dry-run JSON by default.
11. Create drafts only when explicitly requested by the user.
12. Stop after draft creation. Do not publish drafts from follow-ups such as `go ahead`, `continue`, or `post it to Gerrit`.
13. Publish actual comments or existing drafts only after publish-specific intent and a second confirmation.

## Account Configuration

Default config path is a separate file inside the skill:

```text
config/account.json
```

Distribute an empty `config/account.json` with the skill. It must not contain real user, password, or base URL values.

Store Gerrit account and REST draft settings in `account.user`, `account.password`, and `account.base_url` in the skill-local config file. `account.user` is shared by SSH and REST; do not store separate `ssh.user` or `rest.user` fields.

Do not store SSH `host`, SSH `port`, Gerrit project, or package in the config file. Gerrit query infers SSH host and port from the local repo's `git remote -v`; use `scripts/gerrit_query.py --host <host> --port <port>` only for explicit overrides.

An extra path in `account.base_url` is the Gerrit REST context path. It is not a project/package setting, so keep it when that Gerrit server serves REST under that path. This value is redacted from normal output.

```json
{
  "account": {
    "user": "<gerrit-user>",
    "password": "<HTTP_PASSWORD>",
    "base_url": "<gerrit-rest-base-url>"
  }
}
```

Use another config file only when explicitly needed with `--config`.

```bash
scripts/configure_account.py --config /path/to/account.json --show
```

Create or update the account config interactively.

```bash
scripts/configure_account.py
```

Enter the Gerrit user, REST base URL, and password or auth header only at the prompts. Do not put account values, passwords, base URLs, or auth headers in shell command arguments.

If values are already configured, run the same command to update only the fields that need to change:

```bash
scripts/configure_account.py
```

Show the current config.

```bash
scripts/configure_account.py --show
```

## Scripts

### `scripts/gerrit_query.py`

Runs Gerrit SSH query and normalizes the result.

```bash
scripts/configure_account.py --show
scripts/gerrit_query.py --diagnose
scripts/gerrit_query.py --change 1837786
scripts/gerrit_query.py --query change:I35167bb952735969c2ac4ab1e55044e80649c580
```

The output includes local git context, Gerrit change metadata, current patch set, approvals, reviewers, change-level messages, inline comments, and whether local `HEAD` matches the current patch set.

### `scripts/build_review_input.py`

Converts reply draft JSON into Gerrit `ReviewInput` JSON.
It automatically prefixes every comment message and change-level review message with `[Codex로 생성됨]`.

```bash
scripts/build_review_input.py replies.json --message "Review analysis dry-run"
```

Defaults:

- `notify: NONE`
- `omit_duplicate_comments: true`
- `tag: autogenerated:codex-gerrit-review`

### `scripts/create_draft_comments.py`

Builds REST draft comment requests. The default mode is dry-run. Real creation happens only with `--execute`.
It automatically prefixes every draft comment message with `[Codex로 생성됨]`.

```bash
scripts/create_draft_comments.py replies.json \
  --change 12345 \
  --revision 3
```

Actual draft creation requires REST authentication fields in `config/account.json` and `--execute`.

### `scripts/publish_drafts.py`

Publishes already-created drafts as real comments. The default mode is dry-run.

```bash
scripts/publish_drafts.py --change 12345 --revision 3
scripts/publish_drafts.py --change 12345 --revision 3 \
  --execute --confirm PUBLISH_DRAFTS:12345,3
```

Use this script only after the user explicitly asks to publish existing drafts and provides a second confirmation for the exact target.

## Safety Rules

- Do not post anything to Gerrit unless the user explicitly requests it.
- Stop and print setup guidance if Gerrit account config is missing.
- Default to analysis results and dry-run JSON.
- Draft creation is not publishing. Stop after draft creation when the user asked for drafts.
- Do not treat generic continuation phrases as publish approval.
- Require publish-specific intent and second confirmation before actual publishing.
- Exclude `submit`, `abandon`, `restore`, `rebase`, `move`, and label votes from the default skill behavior.
- Restrict real publishing to comments only.
- Do not publish comments when local `HEAD` differs from the Gerrit current patch set revision.
- Require concrete code file and line evidence before classifying a comment as `false_positive`.
- Use `needs_more_context` instead of `false_positive` when the evidence is incomplete.

## Output Format

Review analysis should usually be organized in this order.

- Summary
- Patch set information
- Comment verdicts
- False positives
- Valid issues
- Gerrit reply drafts
- Dry-run JSON
- Posting status

For this user's repositories, final answers should be split into `사실` and `추측`.
